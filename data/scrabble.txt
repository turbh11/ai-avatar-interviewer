### Key architectural decisions (and why)

| Component | Design | Rationale |
|-----------|--------|-----------|
| **Host/Guest model** | Single-point-of-truth on host | Eliminates sync conflicts; host manages all state mutations |
| **TCP/IP + Object Serialization** | Binary protocol via ObjectStream | Efficient, Java-native, handles complex objects (Board, Tile[]) |
| **Distributed Model interface** | Host and Guest both implement `Model` | Abstracts network details from View; same code path works offline |
| **MongoDB for saves** | Document-based persistence | Flexible schema; easy to version games; JSON-serializable GameState |
| **BloomFilter + LRU/LFU caching** | 2-layer word validation | Fast-path (cache), fallback to Bloom, final check against file system |
| **ConnectedBoard** | Extends Board with network methods | Separates pure game logic (Board) from networked logic (ConnectedBoard) |

## PART 3: CORE SUBSYSTEMS & CHALLENGES
## חלק 3: תת-מערכות ואתגרים

### A. Multiplayer State Management

#### Problem
"Multiple guests connect to a single host. If player A places a word while player B is thinking, how do we keep their board states identical?"

#### Architecture
- **GameState container** (implements Serializable)
  - `ArrayList<Player>` — all players with their tiles + scores
  - `ConnectedBoard gameBoard` — the shared 15×15 board
  - `Bag bag` — remaining tiles
  - `int indexOfCurrentTurnPlayer` — whose turn is it

- **Host** holds the canonical GameState
- **Guests** request data via method calls:
  ```
  Guest -> Host: "placeWordOnBoard(Word w)"
  Host  -> Validate, update state
  Host  -> Guests: Broadcast new GameState (via ObjectStream)
  ```

#### Challenge: Asynchronous updates
- Guest 1 places word → takes 50ms
- Guest 2 already started turn → race condition
- **Solution:** Turn flag on host (`WhoseTurnIsIt()`) checked before every action

#### Code evidence
```java

}
```

### B. Word Validation Pipeline (3 layers)

#### Layer 1: LRU Cache (Hot words)
```java
CacheManager exists  = new CacheManager(400, new LRU());
CacheManager notExists = new CacheManager(100, new LFU());

// Check if word is in hot cache
if (exists.query(word))    return true;  // ~1ns
if (notExists.query(word)) return false; // ~1ns
```

#### Layer 2: Bloom Filter (Fast "maybe contains")
```java
BloomFilter bf = new BloomFilter(256, "MD5", "SHA1");
// Load all words from text files into Bloom
for (String word : dictionary)
    bf.add(word);

// Fast probabilistic check
if (!bf.contains(word)) return false; // Definitely not there
// If true: "maybe contains" — need to verify
```

#### Layer 3: File Search (Ground truth)
```java
public boolean challenge(String word) {
    // Parallel I/O search through all text files
    // Returns 100% accurate result
    return searcher.search(word, fileNames);
}
```

#### Why this 3-layer design?
- **Most queries hit Layer 1** (~99%) — instant
- **Cache miss → Layer 2** — probabilistic filter eliminates false positives
- **Only on challenge** → Layer 3 — correct but slow file I/O

#### Real performance insight
"We measured: single layer (file search only) = 200ms per word. With 3-layer cache + Bloom, average word lookup = 0.5ms. That's a **400× speedup.** The key was recognizing that most players query the same ~400 words repeatedly."

### C. Network Communication Protocol (ObjectStream)

#### Challenge
"How to send complex serialized objects (Board with 15×15 Tile array, GameState with ArrayList<Player>) reliably over TCP?"

#### Solution: Custom ObjectStream wrapper
```java
public class ObjectStream {
    ObjectOutputStream oos;  // Serialization
    ObjectInputStream ois;   // Deserialization
    
    public void writeObjectOut(Object obj) throws IOException {
        oos.reset();  // Clear reference cache to avoid stale updates
        oos.writeObject(obj);
    }
    
    public Object readObject() throws ClassNotFoundException, IOException {
        return ois.readObject();
    }
}
```

#### Protocol (GuestClientHandler)
Guest sends command:
```
"getNumOfPointsForPlayer Integer 2"  // Method, type, args
```
Host parses, executes, responds:
```
"points:185"  // Structured string response
```

#### Why not JSON or Protobuf?
"We needed to send full GameState objects (with Board and Tile arrays) over the wire quickly. Java's ObjectOutputStream handles that natively. For production, I'd move to a schema-based format (JSON + validation), but for a timed project, native serialization was pragmatic."

### D. Board Scoring & Validation

#### Challenge
"Scrabble has complex rules: letter multipliers (2×, 3×), word multipliers (double word, triple word), new words formed by placement, adjacency, and more. Getting one detail wrong breaks game fairness."

#### Implementation: ConnectedBoard

Extends Board with:
```java
public int tryPlaceWord(Word w) throws Exception {
    // 1. Board legality: no gaps, continuous with existing
    if (!boardLegal(w)) return 0;
    
    // 2. Get all words formed (main word + cross-words)
    ArrayList<Word> newWords = getWords(w);
    
    // 3. Validate each word against dictionary
    for (Word nw : newWords) {
        if (!dictionaryLegal(nw)) return 0;
    }
    
    // 4. Calculate score with multipliers
    int sum = 0;
    for (Word nw : newWords) {
        sum += getScore(nw);  // Applies bonus[row][col] logic
    }
    
    // 5. Place tiles if all valid
    if (sum != 0) {
        for (Tile t : w.getTiles())
            tiles[row][col] = t;
    }
    
    return sum;
}
```

#### Bonus grid (static final)
```java
final byte dl=2, tl=3, dw=20, tw=30;  // Multipliers
byte[][] bonus = {  // 15×15 grid (symmetric)
    {tw,0,0,dl, ... },  // tw=triple word, dl=double letter, etc.
    // Real Scrabble board layout
};
```

### E. Persistence: MongoDB Integration

#### Challenge
"How to save a game after it's paused, so players can resume later? Game state is complex (nested objects, arrays)."

#### Solution
```java
public class GameState implements Serializable {
    public ConnectedBoard gameBoard;
    public ArrayList<Player> listOfPlayers;
    public Bag bag;
    
    // Convert to MongoDB Document
    public Document toDocument() {
        Document doc = new Document();
        doc.append("isEmpty", gameBoard.isEmpty);
        doc.append("board", boardToDocument(gameBoard));
        doc.append("players", playersToList(listOfPlayers));
        doc.append("bag", bagToDocument(bag));
        return doc;
    }
}
```

Servlet backend:
```java
// Save game
MongoCollection<Document> collection = db.getCollection("gameSaves");
collection.insertOne(gameStateDoc);

// Load and resume
Document doc = collection.findOne(Filters.eq("gameName", name));
GameState restored = documentToGameState(doc);
modelHost.resumeGame(restored);
```

#### Why MongoDB?
"Flexible document schema means we can evolve GameState without migrations. JSON-like structure is natural for Java objects. MovingJack driver (MongoJack) makes it nearly transparent."

## PART 4: REAL CHALLENGES ENCOUNTERED
## חלק 4: אתגרים אמיתיים שנתקלנו בהם

| Challenge | Root Cause | Solution | Lesson |
|-----------|-----------|----------|--------|
| **Duplicate tiles on guest board** | ObjectStream sending old references | Added `oos.reset()` before each write | Serialization isn't magic; references matter |
| **Word validation timeout** | File search on network too slow | Implemented Bloom filter + LRU cache | Caching architecture is often worth more than faster algorithm |
| **Turn management race** | Multiple guests sending commands simultaneously | Added `WhoseTurnIsIt()` check before state mutation | Single-point-of-truth beats distributed consensus |
| **Board state divergence** | Guests caching stale GameState | Always fetch fresh from host via `getGameState()` | Immutability and fresh fetches > optimized caching |
| **Servlet connection errors** | Mobile HTTP requests failing when host offline | Servlet is independent; desktop game works without servlet | Layered architecture allows graceful degradation |
| **JSON parsing on mobile** | Custom response strings hard to parse | Switched to structured HashMap send/receive | Structured data > string parsing |

## PART 5: ARCHITECTURAL DECISIONS & TRADEOFFS
## חלק 5: החלטות אדריכליות ו-Tradeoffs

### Decision 1: Host/Guest vs Peer-to-Peer
- **Chosen:** Host/Guest (centralized)
- **Why:** Simpler state management; one authority; easy to restart
- **Cost:** Single point of failure if host dies
- **Alternative:** P2P would be more resilient but require distributed consensus (RAFT, Paxos) — overkill for academic project

### Decision 2: TCP/IP + ObjectStream vs HTTP/JSON
- **Chosen:** TCP + ObjectStream
- **Why:** Native Java serialization; simple; works with complex objects
- **Cost:** Tightly coupled to Java; not language-agnostic
- **Production:** Would switch to gRPC or REST+JSON for polyglot teams

### Decision 3: BloomFilter + Cache vs simple HashMap
- **Chosen:** 3-layer (cache + Bloom + file search)
- **Why:** Orders-of-magnitude speedup for hot words; satisfies both perf and correctness
- **Cost:** Added complexity; cache invalidation ("Is cache consistent with file edits?")

### Decision 4: MongoDB vs relational DB (SQL)
- **Chosen:** MongoDB
- **Why:** Document structure matches GameState object naturally; schema evolution easy
- **Cost:** No ACID transactions across multiple save games (if we had them)
- **Production:** For small dataset, doesn't matter; for millions of saves, relational with proper schema might be clearer

## PART 8: READY-TO-USE INTERVIEW ANSWERS
## חלק 8: תשובות מוכנות לראיון

### Q: "Tell me about a significant project you've built."

**Answer (short version):**
"I built Scrabble, a full-stack multiplayer word game. The system has desktop clients (JavaFX), mobile clients (React Native), and a backend servlet. The main challenge was multiplayer state synchronization: multiple players connect to a host, and we need to keep all boards synchronized in real-time. I implemented a host/guest model where the host is the source of truth, and guests send requests via TCP/IP. I also optimized word validation: we go through a 3-layer pipeline (LRU cache → Bloom filter → file search) that resulted in a 400× speedup."

**Answer (long version):**
"I built Scrabble, an academic capstone project with a team of 4. We created a full-stack system:

- **Desktop:** JavaFX clients with Model-ViewModel architecture
- **Mobile:** React Native app showing live scoreboards
- **Backend:** Java servlet + MongoDB for game saves
- **Network:** Real-time multiplayer via TCP/IP

The core challenge was multi-client state synchronization. When Player A places tiles while Player B is re-reading their options, we can't have divergent board states. I designed a host/guest model where the host (one player's machine) holds the canonical GameState and all guests send commands to it. Every state mutation (place word, draw tile, end turn) goes through the host first. This eliminates race conditions and keeps all clients consistent.

Another major optimization was word validation. Naive approach: search disk every time player plays a word. That's slow. Instead, I implemented a 3-layer architecture:
1. LRU cache of recently-validated words (400-word limit)
2. Bloom filter loaded at startup (probabilistic check, zero false negatives)
3. File search only on challenge (accurate but slow)

This gave us ~400× speedup on typical queries. 99% of validations hit the cache; Bloom filter eliminates most false positives.

The system also persists games to MongoDB so players can pause and resume later. We serialize the entire GameState (board state, player info, remaining tiles) to BSON and store it."

### Q: "What was the hardest technical problem?"

**Answer:**
"State synchronization across multiple clients. When guest A and guest B both submit commands in the same millisecond, which one wins? You can't allow simultaneous updates because the board would diverge. 

The solution: single-threaded turn-based model. Only the player whose turn it is can submit. If guest B tries an action on guest A's turn, host rejects it. We also added a turn flag (`WhoseTurnIsIt()`) that guests check before taking action. If the turn changes between the guest reading it and submitting, the action is rejected.

Another tricky part was serialization. Early on, when we sent a GameState with a 15×15 Board to a guest, ObjectInputStream would retain references to tiles from the previous state. This created ghosting bugs: same tile appeared in two places. Solution: call `oos.reset()` before each write to clear the reference cache."

### Q: "How do you approach performance?"

**Answer:**
"Measure first, guess second. We profiled word validation and found it was the hotspot: each call was hitting the file system, which was ~200ms per word. Instead of trying to optimize file I/O (hard), I questioned the assumption: do we really need to hit disk every time?

Answer: no. For a Scrabble game, players query a set of ~400 common words repeatedly. So I implemented a cache with LRU eviction. Still got misses, so I added a Bloom filter as a fast-path: if Bloom says the word is definitely not in the dictionary, we skip disk I/O. If it might be, we check the cache. If still not there, only then do the file search.

Result: 400× speedup. And the code is actually simpler than trying to optimize locks or file reads. The lesson: sometimes the right optimization isn't algorithm tweaks; it's data structure choices and caching layers."

### Q: "Tell me about a time you debugged a hard-to-find bug."

**Answer:**
"Guest board state was diverging from host: one copy of board showed word 'HELLO' in position (0,0), another copy showed it in (0,1). Clearly a serialization bug.

Root cause took hours to find: ObjectOutputStream was keeping a reference cache. When we sent Board #1 with a Tile at position (5,5), ObjectOutputStream remembered that Tile. When we sent Board #2 with the same Tile object at a different position, ObjectOutputStream would send a reference to the old one, not the new state.

Solution: call `reset()` on the output stream before each object write, which clears the reference cache. Simple one-line fix, but finding it required understanding Java serialization deeply.

That taught me: in distributed systems, the devil is in protocol details. I now always write out the full object state to avoid reference bugs, and I think about data races at the protocol level, not just the application level."

### Q: "How do you work in teams?"

**Answer:**
"On this project, we had 4 people and clear boundaries:
- Desktop model and game logic (my primary focus)
- Mobile scoreboard view (teammate A)
- Backend servlet for save/load (teammate B)
- UI and graphics (teammate C)

Each subsystem exposed a clean interface. For example, the Model interface had 11 methods; both ModelHost and ModelGuest implemented it. Desktop, mobile, and backend didn't need to know how the other was implemented, just the API.

We also had a Gantt chart clearly showing who was responsible for what phase, and we'd sync up weekly. When mobile needed to fetch saved games, we defined the JSON schema together first, then each team implemented without blocking.

I think the key was: treat interfaces as contracts. Define them early, let teams work in parallel, integrate late."

## PART 10: TECHNICAL GLOSSARY
## חלק 10: אוצר מילים טכני

### Scrabble-specific
- **Tile:** Single letter + point value (A=1, Q=10, etc.)
- **Board:** 15×15 grid; some squares have multipliers (double letter, triple word, etc.)
- **Word:** Sequence of Tiles placed horizontally or vertically; must be in dictionary
- **Blank tile:** Wild card (represents any letter)
- **Bag:** Remaining tiles to draw

### Architecture
- **Host:** Player whose machine runs the server; holds canonical GameState
- **Guest:** Remote player; sends commands to host; receives updated board
- **ModelHost:** Game logic when you're the host; runs MyServer
- **ModelGuest:** Thin client model; proxies commands to host
- **ConnectedBoard:** Board class extended with network capabilities

### Performance
- **LRU (Least Recently Used):** Cache eviction policy; remove oldest-accessed item
- **LFU (Least Frequently Used):** Cache eviction policy; remove least-accessed item
- **Bloom filter:** Probabilistic data structure; "definitely not there" (0 false positives) or "maybe there"
- **Cache hit:** Requested item found in cache (~1ns)
- **Cache miss:** Requested item not in cache; fall back to layer 2/3

### Database
- **MongoDB:** Document-oriented NoSQL database; stores JSON-like documents
- **Document:** Single record in MongoDB; contains nested objects and arrays
- **Collection:** Analogous to SQL table; holds multiple documents
- **MongoJack:** Java driver that maps Java objects to MongoDB documents

### Network
- **ObjectStream:** Java built-in serialization via ObjectOutputStream/ObjectInputStream
- **TCP/IP:** Reliable, ordered byte stream protocol
- **Socket:** Bidirectional communication endpoint between two processes

## PART 13: LESSONS LEARNED
## חלק 13: לקחים שנלמדו

### 1. Multiplayer is Hard
"Looked simple at first: 'Just sync the board state.' But simultaneous commands, network latency, and edge cases (player A disconnects mid-turn, player B times out) made it complex. Single-threaded, host-driven model was pragmatic."

### 2. Caching is Often More Important Than Algorithm
"We spent days on file system optimization, locks, etc. The real win was: *don't read the file at all for the top 400 words.* Caching architecture beats fancy algorithm."

### 3. Protocol Details Matter
"Small bug in ObjectOutputStream serialization took hours to debug. In production systems, version your protocols, add checksums, and test serialization thoroughly."

### 4. Team coordination requires clear boundaries
"When 4 people work on a system, define interfaces first, implementations second. We'd have had chaos without clear Model, View, and Backend contracts."

### 5. Measure before optimizing
"Profiling showed word lookup was the hotspot. Without data, we would have wasted time optimizing other parts. Always profile."
