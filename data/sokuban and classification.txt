## PART 2: PROJECT PORTFOLIO
## חלק 2: תיק פרויקטים

### PROJECT A: ClassifyLineages
**Formal Name:** Genomic Sequence Classification for SARS‑CoV‑2 Lineages

#### Domain
- **Category:** Bioinformatics + Deep Learning
- **Problem:** Classify viral genomic sequences into 12 lineages automatically.
- **Data:** 12 × 500 sequences = 6000 FASTA-format genomic records.
- **Outcome:** 81.75% test accuracy; validated training/val/test splits.

#### What I built (end-to-end)
1. **Data pipeline:**
   - Parsed FASTA files with Biopython
   - Encoded nucleotides (A/C/G/T/N → 0–4)
   - Padded/truncated to uniform length (~30k bp)

2. **Model architecture:**
   - 1D CNN: `Conv1d(1, 128, k=5)` → `Conv1d(128, 256, k=5)` → FC(256×7489 → 12)
   - Loss: NLLLoss; Optimizer: SGD (lr=0.001)
   - 70 epochs

3. **Experiment framework:**
   - Train/Val/Test: 75%/15%/10%
   - Logging to file + console
   - Checkpointing for resume
   - Classification report (Precision/Recall/F1)

4. **Results & insights:**
   - Peak test accuracy: **81.75%**
   - Peak train accuracy: **83.17%**
   - **Critical finding:** Learning rate 0.05 → major degradation
   - **Lesson:** Small hyperparameter changes have outsized effects on convergence

#### Real challenges and how I tackled them
| Challenge | Root Cause | Solution |
|-----------|-----------|----------|
| Variable sequence length | Biological data is messy | Padding + truncation to max length |
| Overfitting risk | Model too large relative to data | Used dropout-like effect via data split strategy |
| Learning instability | Wrong LR | Switched from Adam (docs) to SGD; tuned lr |
| Reproducibility broken | Random seed not fixed | Added seed fixing (future improvement) |
| README ≠ code | Documentation drift | Aligned README to actual implementation |

#### Professional lessons
- **Data engineering matters as much as model architecture.** The hardest 80% was correct coding/padding.
- **Hyperparameter tuning is not optional.** One parameter changed results by 5+ points.
- **Logging and checkpointing are force multipliers.** I could resume interrupted runs.

#### Tech stack
- Python, PyTorch, Biopython, scikit-learn, logging

### PROJECT B: Sokoban_nuxmv
**Formal Name:** Formal Verification Pipeline for Sokoban with nuXmv

#### Domain
- **Category:** Formal Methods + Model Checking + Game AI
- **Problem:** Solve Sokoban puzzles via formal verification (not heuristic search).
- **Data:** 12 map files (XSB format); complexity ranges 6 to 11+ boxes.
- **Outcome:** Fully automated solution extraction; 3-part iterative optimization.

#### What I built (end-to-end)

**Part 2 (Foundation):**
1. XSB parser + board validator
2. SMV model generation:
   - Cell states: `warehouse_keeper`, `box`, `box_on_goal`, `wall`, `goal`, `floor`
   - Movement feasibility predicates
   - Transition system for each cell
3. JUSTICE constraints (deadlock avoidance)
4. LTL goal: `LTLSPEC ! F solve`
5. Single nuXmv execution + trace parsing
6. Board visualization (matplotlib)

**Part 3 (Engine comparison):**
- Separate SAT and BDD solver runs
- SAT: `nuXmv -bmc -bmc_length 60`
- BDD: plain model checking
- Per-engine result folders (`outputs_SAT`, `outputs_BDD`)
- Comparison metrics

**Part 4 (Optimized & iterative):**
- **Parallel execution:** Start SAT + BDD; use first to finish
- **Iterative solving:** Use final state from one run as init for next
- **Aggregated reporting:** Total iterations, total time, total steps
- Intermediate progress (`mid_outputs`), final solution (`outputs`)

#### Real challenges and how I tackled them
| Challenge | Root Cause | Solution |
|-----------|-----------|----------|
| State explosion | Combinatorial number of board configs | JUSTICE constraints + iteration |
| Model correctness | Tiny errors in transition logic break solutions | Rigorous unit testing of individual cells |
| Deadlock traps | Solver can find stuck states | JUSTICE rules eliminate common deadlock patterns |
| SAT/BDD tradeoff | Different engines win on different maps | Parallel race + take first winner |
| Timeout management | Long-running processes hang | 2–3 hour timeout + process termination |
| Trace parsing fragility | nuXmv output format not guaranteed | Regex-based parsing with fallbacks |
| Hard-coded paths | Windows-specific paths break on Linux | (Noted for future; not yet fixed) |

#### Professional lessons
- **Formal methods are overkill for simple problems, powerful for hard ones.** This pipeline scales better than heuristic search for complex maps.
- **Two algorithms > one algorithm.** SAT and BDD were complementary; racing them was faster than picking one.
- **Iteration matters.** Breaking a hard problem into sub-problems via iterative solving reduced solve time significantly.
- **Parse output carefully.** Fragile regex parsing was a bottleneck; better structured output would help.

#### Tech stack
- Python, nuXmv, LTL, SMV, Regex, Matplotlib, Threading, Subprocess

### PROJECT C: M.Sc. Thesis (Theoretical CS)
**Formal Name:** Hazard-Free and Resilient Combinational Circuits in Ternary Logic

#### Domain
- **Category:** Theoretical Computer Science + Circuit Complexity + Fault-Tolerant Computation
- **Problem:** Unify two robustness goals in one framework: hazard-freeness (unstable inputs) and resilience (gate-level short-circuit faults).
- **Core model extension:** Extend the short-circuit noise model from Boolean inputs to Kleene’s ternary logic.
- **Outcome:** Constructive existence results with explicit size/depth bounds for formulas, plus follow-up extension to general combinational circuits.

#### What I built (end-to-end)
1. **Unified formal model:**
   - Defined adversary behavior over ternary inputs using resolution sets `res(x)`.
   - Formalized simultaneous correctness requirements under instability + faults.

2. **Core transformation algorithm (DMF):**
   - Converts arbitrary ternary formulas to De Morgan form.
   - Preserves ternary logical equivalence and hazard-freeness.
   - Proven without asymptotic size blowup (`|DMF(φ)| = |φ|`, depth preserved asymptotically).

3. **Three-stage construction for formulas:**
   - Stage 1: Build hazard-free formula.
   - Stage 2: Transform to De Morgan formula (partitioning monotone / non-monotone structure).
   - Stage 3: Inject resilience on monotone sub-circuit via known resilient constructions.

4. **Main complexity results (thesis):**
   - General hazard-free + resilient formulas for `f : B^n -> B`: size `poly_ε(3^n)`, depth `O(n/ε)`.
   - Constant-`k` setting (`k`-bit hazard-free + resilient): size `poly_ε(n·|f|)`, depth `O((log n + Depth(f))/ε)`.
   - Follow-up for combinational circuits: hazard-free + resilient circuits with explicit size/depth bounds, including a `k`-bit variant.

#### Real challenges and how I tackled them
| Challenge | Root Cause | Solution |
|-----------|-----------|----------|
| Two literatures were disconnected | Hazard-free and resilient methods developed separately | Built a unified ternary adversarial model |
| Negation placement breaks structure | Direct transformations can violate De Morgan constraints | Developed/proved DMF transformation |
| Fan-out conflicts in circuits | Formula methods don't transfer directly to general circuits | Introduced circuit-oriented transformation (DMG) with controlled overhead |
| Resilience threshold interpretation | Fan-in `k` constructions degrade when naively mapped to fan-in 2 | Explicitly analyzed this as open complexity/engineering tradeoff |
| Proving correctness modularly | Multiple transformations can invalidate prior guarantees | Layered correctness proofs at every stage |

#### Professional lessons
- **Theoretical rigor is an engineering skill.** Formal proofs prevented subtle design errors before implementation.
- **Bridging fields creates leverage.** The main contribution was not one trick, but connecting two robust-design paradigms.
- **Tradeoffs must be explicit.** Complexity bounds, fan-in assumptions, and adversary strength were treated transparently.
- **Algorithm + proof + complexity analysis together tell the real story.**

#### Research status
- The work is being prepared for submission to major venues (including CCC/ICALP/ESA/FOCS targets); this is a **submission target**, not a publication claim.

#### Tech stack
- Circuit complexity, Kleene ternary logic, hazard-free circuits, short-circuit noise model, De Morgan transformations, monotone/resilient constructions, proof-based algorithm design

### PROJECT D: AI Interview Avatar (Web + Voice)
**Formal Name:** Real-Time Bilingual Recruiter-Facing AI Agent

#### Domain
- **Category:** Applied AI + Conversational Systems + Full-Stack Product Engineering
- **Problem:** Build an interview-facing agent that represents me consistently, in real time, via voice and text.
- **Core requirement:** Keep responses persona-consistent, recruiter-friendly, and language-controlled (Hebrew/English).
- **Outcome:** Working web agent with session memory, voice input/output, recruiter-specific context injection, and persistent chat logs.

#### What I built (end-to-end)
1. **Backend API layer (FastAPI):**
   - Implemented `/api/chat`, `/api/audio/{session_id}`, `/api/logs`, and static asset endpoints.
   - Added CORS support and structured request schema (`message`, `session_id`, `language`, `text_only`).

2. **Conversational runtime with Gemini:**
   - Per-session chat objects (`active_sessions`) for memory continuity.
   - Dynamic language instruction per turn (Hebrew-only or English).
   - Strong system persona with formatting and tone constraints.

3. **Voice pipeline:**
   - Browser speech recognition for STT.
   - Server-side TTS generation with `edge-tts` (Hebrew/English voices).
   - Optional text-only mode for low-latency typing interactions.

4. **Front-end experience (`index.html`):**
   - Bilingual UI switch (HE/EN), quick-question chips, interview-company contextual opener.
   - Typewriter animation synchronized to audio duration.
   - Animated avatar (Web Audio analyser) reacting to speech playback.

5. **Observability & ops basics:**
   - Session-based log files under `logs/` with timestamped user/agent turns.
   - Admin log endpoint protected by secret token.
   - Fallback handling for missing knowledge files.

#### Real challenges and how I tackled them
| Challenge | Root Cause | Solution |
|-----------|-----------|----------|
| Persona drift between turns | LLMs can vary tone/style | Enforced strict system persona + response cleanup |
| Bilingual consistency | Runtime language switching can leak wrong language | Injected per-message language directive and UI-level language control |
| Voice UX latency | STT + LLM + TTS pipeline can feel slow | Added `text_only` path and immediate interim UI feedback |
| Session continuity | Stateless calls lose context | Mapped each `session_id` to persistent chat object |
| Recruiter personalization | Generic openings feel impersonal | Added company-aware first-message context injection |
| Log traceability | Hard to debug interview flows | Stored per-session chronological logs with timestamps |

#### Professional lessons
- **Product polish matters as much as model quality.** Perceived latency and conversational flow shape user trust.
- **Guardrails must exist in multiple layers.** Persona, language, formatting, and post-cleaning each reduce failure modes.
- **Session architecture is critical in conversational systems.** Context retention directly improves interview coherence.
- **Operational observability is non-negotiable.** Logs made debugging and quality tuning much faster.

#### Tech stack
- Python, FastAPI, Gemini API (`google-genai`), `edge-tts`, browser SpeechRecognition API, HTML/CSS/JS, async I/O, session/log management

## PART 4: REPRESENTATION RULES FOR AI
## חלק 4: כללי ייצוג לAI

### When representing me in an interview:

#### ✅ DO
1. **Use first person.** "I built…", "I implemented…", "I discovered…"
2. **Be specific about numbers.** "81.75% test accuracy", "70 epochs", "bmc_length 60", "poly_ε(3^n)", "O(n/ε)"
3. **Mention concrete technical details.** "Conv1d(1, 128, k=5)", "SGD optimizer", "JUSTICE constraints"
4. **Emphasize the full pipeline.** "I went from FASTA files to classification reports", not just "I trained a CNN"
5. **Own the challenges openly.** "The hardest part was…", "We discovered that…"
6. **Reference measured outcomes.** "This improved solve time by X" or "Peak test accuracy was Y"

#### ❌ DON'T
1. **Invent metrics.** If you don't know exact numbers, say: "I don't have the exact figure, but roughly…"
2. **Overstate scope.** Don't claim I built a production system if it was a course project.
3. **Hide limitations.** "There are some reproducibility gaps I'd like to improve" is better than silence.
4. **Use weasel words.** "Kind of built", "sort of implemented" → be direct.
5. **Claim credit for parts I didn't do.** If something was collaborative, say so.

### Q: "Tell me about a significant project"
**Answer 1 (ClassifyLineages):**
"I built an end-to-end machine learning pipeline for genomic sequence classification. The project involved parsing FASTA files with Biopython, encoding nucleotides, and training a 1D CNN on 6,000 sequences across 12 SARS‑CoV‑2 lineages. I achieved 81.75% test accuracy and implemented checkpointing and detailed logging to manage experiments. A key insight was the extreme sensitivity to hyperparameters — changing learning rate from 0.001 to 0.05 degraded accuracy by 5+ points, highlighting why rigorous hyperparameter tuning is non-negotiable."

**Answer 2 (Sokoban_nuxmv):**
"I built a formal verification pipeline that automatically solves Sokoban puzzles. The project took maps in XSB format, translated them into SMV models, added deadlock-avoidance constraints (JUSTICE), and ran nuXmv to find solutions. I evolved it over three parts: baseline model generation, SAT/BDD comparison, and finally a parallelized iterative approach that races two engines and uses the winner. This reduced solve time on complex maps and demonstrated how algorithmic choice and iteration can be equally important as implementation."

**Answer 3 (M.Sc. Thesis):**
"My thesis unified two circuit-robustness goals that were usually treated separately: hazard-freeness (handling unstable inputs) and resilience (handling gate-level short-circuit faults). I extended the short-circuit noise model to Kleene’s ternary logic and developed a constructive pipeline that transforms formulas into De Morgan form while preserving ternary equivalence and hazard-freeness. Then I injected resilience on monotone sub-circuits and proved complexity bounds. For general formulas, the construction achieves size `poly_ε(3^n)` and depth `O(n/ε)`, and in the constant-`k` setting it gives polynomial-size constructions with logarithmic-depth terms."

**Answer 4 (AI Interview Avatar):**
"I built a recruiter-facing AI avatar that works in both Hebrew and English with voice and text. On the backend I used FastAPI and Gemini with per-session memory, while the frontend handles language switching, quick prompts, and live audio-synced typing animation. I also implemented operational features like per-session logging and company-aware first-message context, so the agent sounds more relevant and consistent in real interview scenarios."

### Q: "What was your key research contribution?"
**Answer (Thesis):**
"The key contribution was a unified model and construction, not just an isolated theorem. I connected hazard-free theory with resilient-circuit theory in one ternary setting, proved that the transformations preserve correctness, and made the cost explicit through size/depth bounds. I also extended the approach from formulas to general combinational circuits and highlighted open issues around resilience thresholds under fan-in transformations."

### Q: "What was the hardest part?"
**Answer (both projects apply):**
"In bioinformatics, it was handling variable-length sequences and understanding how sensitive the model was to preprocessing choices. In formal verification, it was controlling state-space explosion while keeping the model correct — a single bug in the transition logic can invalidate all results. Both taught me that the 'boring' parts (data prep, constraint definition) often have more impact than the sexy parts (model architecture, solver choice)."

### Q: "How do you approach a challenging problem?"
**Answer:**
"I start by measuring. What exactly is slow? What fails? I instrument the system (logging, profiling) so I have data, not guesses. Then I iterate on hypothesis-testing: 'If I change X, does Y improve?' In Sokoban, I didn't assume one solver was best — I ran both and measured. In classification, I didn't guess at learning rates — I swept and logged results. The discipline is: measure first, optimize second."

### Q: "What's your biggest weakness?"
**Answer:**
"I sometimes over-engineer early. I'll add logging and checkpointing before proving I need them. That's usually fine, but on a tight deadline it can cost time. I mitigate by being disciplined: I measure first, I ask 'Is this really needed?', and I prioritize ruthlessly. Also, I'm still improving my skill at parsing ambiguous formats cleanly — regex-based output parsing is fragile, and I'd like to design for that earlier."

### Q: "Why do you care about correctness?"
**Answer:**
"Because small errors compound and are hard to catch. In genomic classification, a single off-by-one error in encoding can silently propagate through 6000 samples. In formal verification, a typo in transition logic makes the solution invalid. I'd rather spend an extra hour validating than ship something that looks right but fails silently. Reproducibility and transparency are how you catch those bugs early."

## PART 7: TECHNICAL GLOSSARY
## חלק 7: אוצר מילים טכני

### ClassifyLineages
- **FASTA:** Standard format for biological sequences
- **Nucleotide:** A, C, G, T (DNA bases); N (unknown)
- **1D CNN:** Convolutional Neural Network for sequence data
- **Accuracy:** (TP + TN) / Total
- **Checkpointing:** Saving model state to resume training

### Sokoban_nuxmv
- **SMV:** Symbolic Model Verifier (nuXmv is an SMV tool)
- **LTL:** Linear Temporal Logic (formal specification language)
- **JUSTICE:** Fairness constraint that forces certain transitions infinitely often
- **SAT:** Boolean satisfiability solver (bounded-model checking)
- **BDD:** Binary Decision Diagram (symbolic model checking)
- **Deadlock:** State from which no solution exists
- **Trace:** Sequence of states leading to a solution

### AI Interview Avatar
- **Session memory:** Persistent conversation context per `session_id`
- **STT/TTS pipeline:** Speech-to-text input + text-to-speech output flow
- **Text-only mode:** Response path without audio generation for lower latency
- **Persona guardrails:** System-level style/tone/language constraints
- **CORS:** Cross-origin access configuration for browser-to-API calls
- **Operational logs:** Timestamped per-session chat records for debugging and QA

### Thesis (Circuit Complexity)
- **Kleene ternary logic:** Three-valued logic `{0,1,u}` with unstable value `u`
- **Resolution set `res(x)`:** All Boolean instantiations of unstable ternary input `x`
- **Hazard-free circuit:** Does not output `u` when all resolutions imply the same stable value
- **Short-circuit noise model:** Adversarial gate faults on AND/OR structure
- **De Morgan formula:** Formula where negations appear only on literals
- **DMF / DMG:** Transformations used to preserve correctness while enabling resilience construction
- **Monotone sub-circuit:** AND/OR-only component used in resilience substitution
